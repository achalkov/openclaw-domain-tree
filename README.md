# openclaw-domain-tree

> Навигационное дерево знаний для агента. Живой документ — растёт по мере того, как агент узнаёт что-то новое.

---

## Для человека

### Зачем это

Когда агент запускается, он не должен читать всё подряд. Он должен уметь найти нужное — как человек ориентируется в незнакомом городе: смотрит на указатели, идёт в нужную сторону, не изучая весь город заранее.

Это дерево — система таких указателей. Оно хранит знания агента: правила, факты, контекст — и позволяет добраться до любого из них по цепочке файлов, не загружая всё сразу.

### Как устроено

Дерево состоит из **узлов**. Каждый узел — это директория с `.md` файлом внутри.

Файл узла делает две вещи:
- Описывает **что это** (правила, факты, контекст для этого уровня)
- Говорит **куда идти дальше** (явно ссылается на дочерние узлы или листы)

Рядом с файлом узла могут лежать **листы** — обычные `.md` файлы без поддиректорий. Это конечные документы: туда идти больше некуда, там просто содержимое.

Глубина дерева — не более 4 уровней. Не потому что так решили заранее, а потому что глубже обычно не нужно.

### Как агент ходит по дереву

Сверху вниз, только по нужной ветке.

Пример: нужно найти информацию о VPN на домашнем сервере.
1. Читает `AGENTS.md` → видит: инфраструктура — иди в `domains/inventory/`
2. Читает `inventory.md` → видит: домашний сервер — иди в `homelab/`
3. Читает `homelab.md` → видит: сеть — иди в `networking/`
4. Читает `networking.md` → видит: VPN — читай `vpn.md`
5. Читает `vpn.md` — нашёл что нужно

Всё остальное дерево не трогалось.

### Как дерево растёт

Дерево не проектируется заранее. Оно отражает то, что агент **уже знает**. Когда появляется новая информация:

- Агент ищет подходящее место в дереве
- Если место есть — дополняет существующий файл
- Если места нет — создаёт новый узел (и все промежуточные, если нужно)
- Обновляет родительский файл: добавляет указатель на новое

Дерево в любой момент неполное — и это нормально. Недостающие ветки появятся, когда агент с ними столкнётся.

## Для реализации (LLM-инструкция)

### Модель узла

Каждый узел дерева — это пара `<name>/` (директория) + `<name>.md` (файл внутри).

Файл узла содержит:
1. **Контекст уровня** — правила, факты, общие принципы, применимые ко всему что внутри
2. **Routing section** — явный список: какой дочерний узел или лист за что отвечает и по какому пути

Формат routing section — произвольный, но должен быть явным. Пример:
```
## Навигация
- Сеть (VPN, DNS, Lancache) → ./networking/
- Программное обеспечение (Docker) → ./software/
- Обслуживание → ./maintenance/
- Железо и хранилище → ./storage/
```

Листовые файлы (без поддиректории) перечисляются так же:
```
- Предпочтения пользователя → ./preferences.md
```

### Правила обхода

При получении задачи:
1. Определить к какому домену относится задача — по `AGENTS.md`
2. Перейти в соответствующий L2 узел, прочитать файл
3. Следовать routing section вглубь до тех пор, пока не достигнут нужный лист или подходящий уровень
4. Не читать файлы за пределами найденной ветки

### Правила роста дерева

**Когда добавлять лист:**
- Тема встречается ≥2 раз ИЛИ содержит нетривиальные правила (не помещается в 2–3 строки)
- Лист создаётся в ближайшем подходящем узле

**Когда создавать новый узел (директорию + роутер):**
- Несколько связанных листов логично группируются под одну абстракцию
- ИЛИ задача не вписывается ни в один существующий узел

**После любого изменения:**
- Обновить routing section родительского файла (добавить ссылку на новое)
- Если создан новый L2 домен — добавить его в реестр в `AGENTS.md`

### Ограничения

- Максимальная глубина: **4 уровня** (L1 = корень, L4 = максимум)
- L4 узлы — только листовые файлы, без поддиректорий
- Узел L2 и выше — всегда **абстракция**, никогда конкретная сущность. Конкретные сущности (конкретная машина, конкретный пользователь) — L3 и глубже
- Роутер узла и его листы живут в одной директории

### Текущая структура

Отражает знания агента на момент последнего коммита. Не является финальной — будет расти.

```
AGENTS.md                         ← L1: глобальные правила + реестр доменов

domains/
  inventory/inventory.md          ← L2: все хосты, общие правила работы с инфрой
    homelab/homelab.md            ← L3: домашний сервер (achalkov-homelab)
      networking/networking.md    ← L4 роутер: сеть
        vpn.md                    ← L4 лист
    vps-nl/vps-nl.md              ← L3: VPS Netherlands main
    vps-lisa/vps-lisa.md          ← L3: VPS Netherlands (Лиза)

  users/users.md                  ← L2: все пользователи, общие правила
    artem/artem.md                ← L3 роутер
    dasha/dasha.md                ← L3 роутер

  agent-ops/agent-ops.md          ← L2: самоуправление агента
    memory/memory.md              ← L3 лист
    scheduling/scheduling.md      ← L3 лист
    skills/skills.md              ← L3 лист

  coding/coding.md                ← L2: разработка
    acp/acp.md                    ← L3 лист
    scripts/scripts.md            ← L3 лист

  content/content.md              ← L2: создание и публикация контента
    cloudtext/cloudtext.md        ← L3 лист
    telegram/telegram.md          ← L3 лист

  knowledge/knowledge.md          ← L2: справочные базы знаний
    history/history.md            ← L3 лист
    science/science.md            ← L3 лист

  finance/finance.md              ← L2 лист (поддоменов пока нет)
```
