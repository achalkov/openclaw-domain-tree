# openclaw-domain-tree

> Навигационное дерево знаний для агента. Живой документ — растёт по мере того, как агент узнаёт что-то новое.

---

## Для человека

### Зачем это

Когда агент запускается, он не должен читать всё подряд. Он должен уметь найти нужное — как человек ориентируется в незнакомом городе: смотрит на указатели, идёт в нужную сторону, не изучая весь город заранее.

Это дерево — система таких указателей. Оно хранит знания агента: правила, факты, контекст — и позволяет добраться до любого из них по цепочке файлов, не загружая всё сразу.

### Как устроено

Дерево состоит из **7 корневых доменов (L1)**. Каждый корневой домен — это пара `<NAME>.md` (L1-файл) + `<NAME>/` (директория с деревом внутри).

L1-файл описывает суть домена и указывает на L2-детей. Всё конкретное — внутри директории.

Рядом с роутером могут лежать **листы** — `.md` файлы без поддиректории. Это конечные документы: там просто содержимое, идти дальше некуда.

### Корневые домены (L1) и лимиты глубины

| Домен | Описание | Макс. глубина |
|-------|----------|---------------|
| `AGENTS.md` + `AGENTS/` | Правила работы агента: инфраструктура, разработка, контент, финансы, самоуправление | L1–L4 |
| `SOUL.md` | Личность агента: тон, манера письма, самоосознание | L1–L2 |
| `TOOLS.md` + `TOOLS/` | Конкретный инструментарий: STT-пайплайн, SSH, принтер и т.п. | L1–L4 |
| `IDENTITY.md` | Паспорт агента: имя, верхнеуровневая модель поведения | L1–L2 |
| `USER.md` + `USER/` | Пользователи агента и вся связанная с ними информация | L1–L3 |
| `HEARTBEAT.md` | Проактивные задачи по расписанию. Плоский файл — без дочерних узлов | L1 |
| `MEMORY.md` | Процедуры доступа к памяти + суперключевые факты. Плоский файл | L1 |

### Как агент ходит по дереву

Сверху вниз, только по нужной ветке.

Пример: нужно найти информацию о VPN на домашнем сервере.
1. Читает `AGENTS.md` → видит: инфраструктура — иди в `AGENTS/inventory/`
2. Читает `AGENTS/inventory/inventory.md` → видит: физические машины — иди в `AGENTS/inventory/baremetal/`
3. Читает `AGENTS/inventory/baremetal/baremetal.md` → видит: домашний сервер — читай `AGENTS/inventory/baremetal/homelab/homelab.md`
4. Читает `homelab.md` → секция VPN — нашёл что нужно

Всё остальное дерево не трогалось.

### Как дерево растёт

Дерево не проектируется заранее. Оно отражает то, что агент **уже знает**. Когда появляется новая информация:

- Агент ищет подходящее место в дереве
- Если место есть — дополняет существующий файл
- Если места нет — создаёт новый узел (и все промежуточные, если нужно)
- Обновляет родительский файл: добавляет указатель на новое

Дерево в любой момент неполное — и это нормально. Недостающие ветки появятся, когда агент с ними столкнётся.

## Для реализации (LLM-инструкция)

### Модель уровней

```
<DOMAIN>.md           ← L1: корневой файл домена (реестр + суть)
<DOMAIN>/
  <l2>.md             ← L2: абстрактный роутер или лист
  <l2>/
    <l3>.md           ← L3: конкретная сущность или лист
    <l3>/
      <l4>.md         ← L4: лист (максимум, без поддиректорий)
```

**Важно:** уровни считаются от L1-файла вниз. `AGENTS.md` — это L1; `AGENTS/inventory/inventory.md` — L2; `AGENTS/inventory/baremetal/baremetal.md` — L3; `AGENTS/inventory/baremetal/homelab/homelab.md` — L4.

### Правило абстракции

- **L1–L2**: всегда абстракция, никогда конкретная сущность
- **L3+**: можно конкретные сущности (конкретная машина, конкретный пользователь)
- **L4**: только листовые файлы, без поддиректорий

### Модель узла

Каждый роутер-узел — это пара `<name>/` (директория) + `<name>.md` (файл внутри).

Файл узла содержит:
1. **Purpose** — что здесь хранится, что сюда НЕ входит
2. **Routing** — явный список дочерних узлов и листов с однострочным правилом для каждого
3. **Decision boundaries** — почему тема X идёт в child1, а не child2 (без разночтений)
4. **Rules & Directives** — правила уровня
5. **Variables** — переменные окружения в dot-path нотации: `agent.<domain>.<key>`
6. **Accumulated rules** — агент дописывает сюда по мере обучения

### Переменные окружения

Все чувствительные/конфигурируемые значения (IP, порты, пути, credentials) хранятся как переменные:
```
agent.inventory.baremetal.homelab.ssh.primary_host
agent.tools.stt.host
agent.users.artem.tg_id
```

Значения живут в `.env`. В `.md`-файлах только имена переменных, не сами значения.

### Правила обхода

При получении задачи:
1. Определить к какому L1-домену относится задача
2. Прочитать L1-файл → следовать routing section
3. Идти вглубь только по нужной ветке до нужного листа
4. Не читать файлы за пределами найденной ветки

### Правила роста дерева

**Когда добавлять лист:**
- Тема встречается ≥2 раз ИЛИ содержит нетривиальные правила (не помещается в 2–3 строки)

**Когда создавать новый узел (директорию + роутер):**
- Несколько связанных листов логично группируются под одну абстракцию
- ИЛИ задача не вписывается ни в один существующий узел

**После любого изменения:**
- Обновить routing section родительского файла
- Если создан новый L2-домен в AGENTS/ — добавить в реестр в `AGENTS.md`
